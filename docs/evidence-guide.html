TL;DR

Evidence = Professional Documentation

Creating quality evidence isn't just about grades‚Äîit's a professional skill. Clear screenshots and organized documentation demonstrate your work effectively and build portfolio-worthy materials.

Essential evidence for Abracadabra

üì∏ Must-have screenshots

  • Terminal commands - Show successful npm run commands with full output
  • Browser magic - Full page screenshot at localhost:3000 showing transformation
  • File structure - VS Code explorer showing generated/ folder contents
  • Package.json - Your custom scripts section clearly visible

Quality standards checklist

  1. Full context visible - Include address bar, terminal prompt, file names
  2. Readable text - Zoom in if necessary, avoid tiny text
  3. Clear file names - Use descriptive names like "step3-npm-run-dev-success.png"
  4. Organized folders - Group by project or assignment

‚ö†Ô∏è Common evidence mistakes

Partial screenshots (missing crucial context), poor file naming (screenshot1.png tells nothing), and wrong timing (taking screenshots after errors occur instead of during success).

Why evidence matters

üéØ Key concept

Evidence-based assessment ensures fair, objective grading by requiring visual proof that requirements were completed correctly. It also builds professional documentation skills.

Benefits for students

  • Clear expectations: Know exactly what needs to be demonstrated
  • Objective grading: Your work speaks for itself through evidence
  • Portfolio building: Screenshots document your learning journey
  • Debugging help: Evidence shows exactly what's happening
  • Professional practice: Learn to document work like developers do

Real-world relevance

Professional scenarios requiring evidence

  • Bug reports: Developers include screenshots to show issues
  • Feature demos: Product managers document new functionality
  • Code reviews: Teams share screenshots of test results
  • Client presentations: Show progress and completed work
  • Documentation: Technical writers create visual guides

üí° Building professional habits

Learning to create clear, comprehensive documentation is a valuable professional skill. The habits you build now will serve you throughout your career.

What makes good evidence

  • Complete context: Shows the full desktop/browser window
  • Clear verification: URLs, file paths, command outputs visible
  • High quality: Readable text and clear interface elements
  • Relevant content: Focused on what needs to be demonstrated
  • Honest representation: Shows actual work, not staged examples

Screenshot standards

Consistent, high-quality screenshots make evidence review efficient and fair.

Technical requirements

Resolution and format

  • Format: PNG (preferred) or JPG
  • Resolution: High enough that text is clearly readable
  • Full screen: Capture entire desktop, not just application windows
  • File size: Balance quality with reasonable file sizes

Content requirements

üéØ Essential elements to include

  • Browser address bar: Shows localhost:3000 for Abracadabra magic
  • VS Code interface: When showing code or terminal work
  • File paths: Visible in VS Code Explorer or terminal
  • Command output: Full terminal output for npm commands
  • Relevant interface elements: Menus, dialogs, status indicators

Clarity standards

‚úÖ Quality checklist

  • All text is clearly readable without zooming
  • Important elements are not obscured
  • Screenshots are well-lit (no glare or shadows)
  • Colors are accurate and not distorted
  • Interface elements are properly aligned (not cropped)

What NOT to include

‚ö†Ô∏è Avoid these common issues

  • Personal information: Close personal files, tabs, notifications
  • Unnecessary content: Social media, games, personal messages
  • Sensitive data: Passwords, API keys, personal documents
  • Cluttered desktop: Too many icons or windows
  • Offensive content: Inappropriate wallpapers, folder names, etc.

What to capture

Different types of evidence serve different purposes in demonstrating completion.

Terminal and command evidence

npm command execution

What to show:

  • The command you typed
  • Complete output including success messages
  • Current directory path visible
  • No error messages (unless part of normal output)

Example scenarios:

  • npm install completing successfully
  • npm run generate creating files
  • npm run serve starting the server

File editing evidence

package.json editing

What to show:

  • VS Code with package.json open
  • File explorer showing correct file is selected
  • Complete scripts section visible
  • Proper JSON syntax with no error indicators
  • File saved (no unsaved dot indicator)

Browser and web evidence

Abracadabra magic activation

What to show:

  • Browser address bar clearly showing localhost:3000
  • Complete webpage with Abracadabra magic activated
  • ASCII art cowsay content visible
  • Farm-themed styling applied
  • No ACME content visible (proves port 3000 worked)

File system evidence

Generated files verification

What to show:

  • File explorer with generated/ folder open
  • All required .txt files present
  • File modification dates showing recent creation
  • Opening one file to show cowsay content

Problem-solving evidence

üí° Documenting challenges

If you encounter and solve problems, consider documenting the solution process. This demonstrates debugging skills and can help other students.

Before and after problem solving

  • Before: Screenshot showing the error or issue

This creates a complete narrative of your problem-solving process

Screenshot tools and techniques

Built-in tools

Windows screenshot tools

  • Snipping Tool: Windows key ‚Üí type "Snipping Tool"
  • Snip & Sketch: Windows key + Shift + S
  • Print Screen: PrtScn key captures full screen
  • Alt + Print Screen: Captures active window only

Mac screenshot tools

  • Full screen: Cmd + Shift + 3
  • Selection: Cmd + Shift + 4
  • Window: Cmd + Shift + 4, then Spacebar
  • Screenshot app: Cmd + Shift + 5 (advanced options)

Browser developer tools

Browser screenshot features

  • Full page screenshot: Chrome DevTools ‚Üí More Tools ‚Üí Capture screenshot
  • Element screenshot: Right-click element ‚Üí Inspect ‚Üí Screenshot node
  • Mobile view screenshot: DevTools responsive mode + screenshot

These capture just the webpage without OS elements

Third-party tools (optional)

Advanced screenshot tools

  • Lightshot: Easy sharing and annotation
  • Greenshot: Open source with editing features
  • Flameshot: Linux/cross-platform tool
  • Monosnap: Mac/PC with cloud integration

Note: Built-in tools are usually sufficient for academic work

Screenshot workflow tips

‚úÖ Efficient screenshot workflow

  1. Prepare your desktop: Close irrelevant windows, arrange workspace
  2. Complete the task: Finish the work before taking screenshots
  3. Take multiple angles: Different views of the same accomplishment
  4. Review immediately: Make sure screenshots are clear and complete
  5. Organize quickly: Move files to correct folders right away
  6. Add descriptions: Brief notes about what each screenshot shows

Common mistakes to avoid

Technical mistakes

‚ùå Partial screenshots

Problem: Cropped images that don't show context

Examples:

  • Browser content without address bar
  • Terminal output without command prompt
  • Code without file name visible

Solution: Always capture full desktop or full application window

‚ùå Poor image quality

Problem: Screenshots that are blurry, too small, or poorly lit

Solutions:

  • Use built-in screenshot tools (not phone photos)
  • Ensure adequate screen brightness
  • Save in high quality format (PNG preferred)
  • Check readability before submitting

Content mistakes

‚ùå Missing verification elements

Common omissions:

  • URL not visible in browser screenshots
  • File paths not shown in VS Code
  • Command prompt not visible in terminal
  • Timestamp or success messages cut off

Impact: Makes it impossible to verify requirements were met

‚ùå Staged or fake evidence

Problem: Evidence that doesn't represent actual work

Examples:

  • Opening example files instead of your actual work
  • Using different ports/URLs than required
  • Showing borrowed or template code as your own

Academic integrity: Evidence should reflect your actual work and process

Organization mistakes

‚ùå Poor file organization

Problems:

  • Screenshots in wrong folders
  • Unclear or duplicate file names
  • Missing required evidence pieces
  • Evidence not organized by specification

Impact: Makes grading difficult and may result in lower scores

Professional documentation tips

Building documentation habits

üéØ Professional mindset

Think of evidence documentation as building a portfolio of your capabilities. Future employers and collaborators will appreciate clear, thorough documentation skills.

Advanced documentation techniques

Annotation and markup

  • Highlight key elements: Circle or arrow to important parts
  • Add brief captions: Explain what the screenshot demonstrates
  • Number sequences: Show order of operations in multi-step processes
  • Use consistent colors: Red for problems, green for success, blue for information

Creating documentation narratives

Tell the story of your work:

  1. Setup: Show initial state and requirements
  2. Process: Document key steps and decisions
  3. Verification: Prove requirements were met
  4. Results: Show final working product

Portfolio building

‚úÖ Long-term value

Well-documented project evidence can become part of your professional portfolio, demonstrating:

  • Technical skills: Ability to work with development tools
  • Problem-solving: Documentation of challenges overcome
  • Attention to detail: Quality of documentation itself
  • Learning progression: Growth from project to project

Collaboration benefits

Documentation for teamwork

Good documentation practices help with:

  • Knowledge sharing: Others can understand your work
  • Troubleshooting: Screenshots help identify and solve problems
  • Process improvement: Document what works and what doesn't
  • Training: Your documentation can help teach others

Continuous improvement

üí° Refining your process

Each project is an opportunity to improve your documentation skills. Reflect on what worked well and what could be clearer for next time.

  • Review your evidence: Before submitting, check clarity and completeness
  • Ask for feedback: Find out what makes evidence most useful
  • Learn from others: Look at examples of clear documentation
  • Develop templates: Create standard approaches for common tasks
  • Practice regularly: Documentation improves with consistent practice
